# Day 9: Encapsulation & Abstraction - Immutable vs Mutable Design

## 📝 Today's Tasks

1.  **Mutable Path:** Implement an object that updates itself via methods.
2.  **Immutable Path:** Implement a version that returns a new instance on change.
3.  **Compare Trade-offs:** Note ease of use, debugging, and sharing across contexts.
4.  **Choose Deliberately:** Document when you'd pick each style in an app.

## 📖 Task Explanations

Mutability is convenient; immutability makes reasoning and concurrency easier. Many modern codebases mix both: immutable values for data, controlled mutability for entities with identity. Understanding both makes you adaptable.

## 💡 What You'll Gain

You will implement mutable and immutable update styles and explain when each fits.

## 📚 Dictionary for Developers

*   **Mutable Object:** An object whose state can change after creation.
*   **Immutable Object:** An object that does not change; updates produce new instances.
*   **Value Object:** An object defined by its values, often immutable (e.g., Money, DateRange).
*   **Entity:** An object with identity that persists over time, even as attributes change.

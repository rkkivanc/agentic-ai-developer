# Day 31: Relations, Lists & Pagination - Modeling One-to-Many Relationships

## 📝 Today's Tasks

1.  **Model Parent and Child Types:** Define `Author` with `posts: [Post!]!` and `Post` with `author: Author!`.
2.  **Choose Cardinality:** Decide optional vs required ends of the relationship with `!` modifiers.
3.  **Foreign Key Fields:** Store `authorId` internally while exposing `author` on the GraphQL type.
4.  **Query Nested Data:** Fetch author with posts in one operation from the client perspective.

## 📖 Task Explanations

One-to-many relationships are the bread and butter of GraphQL APIs. Object-typed fields express relations in the schema while foreign keys stay implementation details in resolvers.

Non-null decisions communicate guarantees—if every post must have an author, mark `author: Author!`; if authors can be deleted, consider nullable author fields.

## 💡 What You'll Gain

By the end of today, you will model one-to-many relationships in SDL and resolve them in nested queries.

## 📚 Dictionary for Developers

*   **One-to-Many:** A parent entity linked to multiple child records.
*   **Foreign Key:** Stored ID referencing a related record in the data layer.
*   **Cardinality:** Whether relationship ends are optional or required.
*   **Nested Query:** A single operation fetching parent and child fields together.
*   **Relation Field:** An object-typed field expressing a link between types.

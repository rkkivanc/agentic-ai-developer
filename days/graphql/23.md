# Day 23: Resolvers & Execution - Resolver Execution Order

## 📝 Today's Tasks

1.  **Log Resolver Order:** Add temporary logs to root and nested field resolvers for one query.
2.  **Understand Depth-First Walk:** See how sibling fields resolve before deeper nested fields.
3.  **Parallel Sibling Fields:** Note that sibling fields at the same depth may resolve concurrently.
4.  **Remove Logging:** Clean up debug logs after tracing to avoid noisy production output.

## 📖 Task Explanations

GraphQL executes resolvers in a depth-first order following the selection set. Understanding this order explains why parent resolvers run before child field resolvers and when parallelization happens.

Tracing execution once demystifies bugs where child resolvers expect parent data that is not yet populated—context and parent values are your friends.

## 💡 What You'll Gain

By the end of today, you will understand GraphQL resolver execution order through hands-on tracing.

## 📚 Dictionary for Developers

*   **Depth-First Execution:** Resolving fields by walking the selection tree top to bottom.
*   **Sibling Fields:** Fields at the same depth under the same parent object.
*   **Concurrent Resolution:** Independent sibling resolvers that may run in parallel.
*   **Execution Engine:** GraphQL-js component orchestrating resolver calls.
*   **Resolver Trace:** Temporary logging revealing call sequence during development.

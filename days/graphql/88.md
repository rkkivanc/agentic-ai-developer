# Day 88: Federation & Multi-Graph - Schema Stitching and Legacy Integration

## 📝 Today's Tasks

1.  **Wrap REST with SDL:** Define GraphQL types mapping to legacy REST JSON shapes.
2.  **Delegate Resolvers:** Forward fields to REST endpoints with transform functions.
3.  **Handle Type Conflicts:** Rename colliding types when merging two schemas.
4.  **Compare to Federation:** Note when stitching fits legacy lift-and-shift versus greenfield federation.

## 📖 Task Explanations

Schema stitching composes multiple schemas or remote executors into one gateway—useful integrating legacy REST services without rewriting them as native GraphQL subgraphs.

Stitching adds operational complexity—prefer native GraphQL services or federation for long-term maintainability when you control the backends.

## 💡 What You'll Gain

By the end of today, you will understand schema stitching for wrapping legacy APIs into a unified GraphQL gateway.

## 📚 Dictionary for Developers

*   **Schema Stitching:** Combining remote schemas or executors into one API surface.
*   **Remote Executor:** Component resolving fields by calling another GraphQL or REST service.
*   **REST Wrapper:** GraphQL resolvers translating to HTTP REST calls underneath.
*   **Type Rename:** Resolving name collisions when merging schemas.
*   **Lift and Shift:** Exposing legacy services through GraphQL without full rewrites.

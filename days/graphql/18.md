# Day 18: Mutations & Inputs - Mutation Payload Patterns

## 📝 Today's Tasks

1.  **Design Payload Types:** Return `CreateItemPayload` with `item` and `errors` fields instead of a bare entity.
2.  **Return Client-Useful Data:** Include fields the UI needs to update lists without a full refetch.
3.  **Study Relay Conventions:** Review optional `clientMutationId` for correlating optimistic updates.
4.  **Sandbox Mutations:** Run create and update mutations and inspect the payload JSON shape.

## 📖 Task Explanations

Payload types make mutation responses predictable and extensible. Room for errors and metadata prevents schema churn when you add validation feedback later.

Apollo Client and urql normalized caches key off returned entity IDs—design payloads so the primary entity includes `id` and `__typename`.

## 💡 What You'll Gain

By the end of today, you will design mutation payload types that support structured errors and efficient client cache updates.

## 📚 Dictionary for Developers

*   **Payload Type:** A mutation return type bundling the affected entity plus metadata.
*   **clientMutationId:** Optional client ID linking optimistic UI state to server results.
*   **Normalized Cache:** Client storage deduplicating objects by type and ID.
*   **Partial Success:** Returning both data and user-facing errors in one response.
*   **Mutation Selection:** Fields requested on the mutation result, kept intentionally small.

# Day 13: Queries & Selection Sets - Variables and Operation Names

## 📝 Today's Tasks

1.  **Declare Query Variables:** Use `$userId: ID!` in operation signatures and pass values via a variables JSON object.
2.  **Name Your Operations:** Label queries like `query GetUser($id: ID!)` for clearer logs and tooling.
3.  **Separate Documents from Variables:** Store static query text in `.graphql` files and variables in client code or sandbox panels.
4.  **Handle Required vs Optional Variables:** Experiment with nullable and non-null variable types.

## 📖 Task Explanations

Variables make queries reusable. Instead of embedding dynamic values in query strings, clients send a variables map alongside the document. This is safer and enables persisted queries later.

Operation names help servers, clients, and analytics identify which query ran. Always name operations in production code.

## 💡 What You'll Gain

By the end of today, you will write parameterized queries with variables and named operations.

## 📚 Dictionary for Developers

*   **Variable Definition:** The `$name: Type` declaration at the top of an operation.
*   **Variables JSON:** A separate object mapping variable names to runtime values.
*   **Literal Value:** A hard-coded value in a query such as `id: "123"` instead of a variable.
*   **Coercion:** Automatic conversion of variable values to the declared GraphQL type.
*   **Operation Signature:** The operation type, name, and variable definitions combined.

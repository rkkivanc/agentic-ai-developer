# Day 61: Schema Design at Scale - Naming Conventions and Schema Consistency

## 📝 Today's Tasks

1.  **Apply Casing Rules:** Use PascalCase types, camelCase fields, and SCREAMING enums.
2.  **Verb Mutations:** Name mutations `createOrder`, not `orderCreate` or `addOrderMutation`.
3.  **Consistent Plurals:** Use `users` for lists and `user` for singles across the schema.
4.  **Lint Naming:** Run GraphQL ESLint naming rules against your SDL files.

## 📖 Task Explanations

Consistent naming reduces cognitive load and improves generated client code. Teams should document conventions before the schema grows unwieldy.

GraphQL ESLint catches drift early—singular list fields and inconsistent mutation verbs are common review nits.

## 💡 What You'll Gain

By the end of today, you will apply consistent GraphQL naming conventions and lint your schema for violations.

## 📚 Dictionary for Developers

*   **PascalCase:** Convention for GraphQL object, input, and interface type names.
*   **camelCase:** Convention for field and argument names.
*   **Mutation Verb:** Leading verb describing the business action being performed.
*   **Pluralization:** Consistent singular/plural naming for list versus item fields.
*   **Naming Lint:** Automated rules flagging non-idiomatic schema identifiers.

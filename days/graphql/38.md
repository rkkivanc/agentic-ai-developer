# Day 38: Errors, Validation & Nullability - Input Validation Strategies

## 📝 Today's Tasks

1.  **Structural vs Business Validation:** Rely on GraphQL for types; implement domain rules in resolvers.
2.  **Collect User Errors:** Return validation messages in mutation payloads instead of throwing for expected failures.
3.  **Validate Relations:** Ensure referenced IDs exist before creating links.
4.  **Test Invalid Inputs:** Run mutations with bad emails, empty strings, and out-of-range numbers.

## 📖 Task Explanations

GraphQL guarantees shape—strings are strings—but not that an email is deliverable or a date is in the future. Business validation belongs in resolver or service layers.

Distinguish validation users can fix from system failures. Expected validation belongs in payload `errors` arrays; database outages belong in logs with generic client messages.

## 💡 What You'll Gain

By the end of today, you will implement business validation with clear user-facing feedback in mutations.

## 📚 Dictionary for Developers

*   **Structural Validation:** Automatic type and required-field checks from GraphQL.
*   **Business Validation:** Domain rules like uniqueness or value ranges in application code.
*   **User Error:** Expected failure such as invalid email safe to show in the UI.
*   **Validation Message:** Human-readable text explaining what input to fix.
*   **Existence Check:** Verifying referenced entities exist before creating relations.

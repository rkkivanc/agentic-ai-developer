# Day 36: Errors, Validation & Nullability - Nullable vs Non-Null Fields

## 📝 Today's Tasks

1.  **Audit Output Nullability:** Review which fields truly always exist versus may be absent.
2.  **Audit Input Nullability:** Mark required mutation inputs with `!` and keep updates optional.
3.  **Observe Cascading Nulls:** Trigger a child resolver error on a non-null field and see parent nulling.
4.  **Document Nullable Choices:** Add descriptions explaining why fields allow null.

## 📖 Task Explanations

Nullability is a contract with clients. Overusing `!` causes entire branches to null when one resolver fails; underusing it forces unnecessary null checks in UI code.

Inputs and outputs have different rules—required create fields differ from partial update inputs where every field is optional.

## 💡 What You'll Gain

By the end of today, you will make deliberate nullability choices and understand cascading null behavior.

## 📚 Dictionary for Developers

*   **Non-null Output:** Field marked `!` that must not be null unless an error bubbles.
*   **Cascading Null:** Parent fields becoming null when a non-null child resolver throws.
*   **Nullable Input:** Optional update field omitted when unchanged.
*   **Nullability Contract:** Schema promises about when null is a valid response value.
*   **Partial Update Input:** Input type where all fields are optional for PATCH-style mutations.

# Day 63: Schema Design at Scale - Deprecation Strategy and Communication

## 📝 Today's Tasks

1.  **Mark `@deprecated`:** Add reason strings pointing to replacement fields on old API surface.
2.  **Set Timelines:** Document when deprecated fields will be removed in changelog entries.
3.  **Track Usage:** Log or metric how often deprecated fields appear in incoming operations.
4.  **Client Outreach:** Draft a migration note for mobile and web teams before removal.

## 📖 Task Explanations

Deprecation is a communication tool, not just an SDL directive. Clients need reasons, replacements, and deadlines to plan migrations.

Never remove fields silently—breaking changes without notice erodes trust and causes production outages.

## 💡 What You'll Gain

By the end of today, you will deprecate schema fields with clear reasons and a communicated removal plan.

## 📚 Dictionary for Developers

*   **@deprecated:** SDL directive marking fields scheduled for removal with a reason.
*   **Migration Guide:** Document helping clients move from old fields to replacements.
*   **Usage Telemetry:** Metrics showing how often deprecated fields are still queried.
*   **Sunset Date:** Planned date when a deprecated field will be removed.
*   **Breaking Removal:** Deleting a field clients still use without a deprecation period.

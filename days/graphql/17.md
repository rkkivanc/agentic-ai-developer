# Day 17: Mutations & Inputs - Input Types and Input Objects

## 📝 Today's Tasks

1.  **Define Input Object Types:** Group related mutation arguments into input types like `CreatePostInput`.
2.  **Mark Required Input Fields:** Use non-null modifiers on fields that must always be provided.
3.  **Avoid Input/Output Type Sharing:** Keep input types separate from object types for evolution flexibility.
4.  **Validate Inputs in Resolvers:** Check business rules after GraphQL's structural validation passes.

## 📖 Task Explanations

Input objects keep mutation signatures clean and evolvable. Adding optional fields to an input type is non-breaking; removing or renaming fields requires deprecation.

GraphQL validates structure automatically, but business validation—unique emails, valid dates—belongs in resolver logic.

## 💡 What You'll Gain

By the end of today, you will design input object types for mutations with appropriate nullability and validation.

## 📚 Dictionary for Developers

*   **Input Object:** A structured input type grouping multiple arguments for mutations.
*   **Side Effect:** A state change on the server caused by a mutation operation.
*   **Payload Type:** A return type wrapping the mutated entity plus metadata like errors or status.
*   **Idempotency:** Property where repeating the same mutation produces the same final state.
*   **Write Operation:** Any GraphQL mutation that creates, updates, or deletes server data.

# Day 62: Schema Design at Scale - Schema Versioning and Evolution

## 📝 Today's Tasks

1.  **Label Your Schema:** Add a `schema { query: Query }` version comment or metadata endpoint.
2.  **Practice Additive Changes:** Add optional fields and new types without breaking existing queries.
3.  **Document Compatibility Rules:** Write team rules for safe vs breaking schema edits.
4.  **Simulate a Breaking Change:** Remove a field locally and observe client validation failures.

## 📖 Task Explanations

GraphQL favors continuous evolution over versioned URLs. Optional new fields are safe; removing or renaming fields breaks clients unless you deprecate first and migrate.

Schema checks in CI compare the new SDL to the last deployed schema—adopt that mindset even on solo projects.

## 💡 What You'll Gain

By the end of today, you will evolve schemas additively and recognize changes that break existing clients.

## 📚 Dictionary for Developers

*   **Additive Change:** A schema update that does not invalidate existing operations.
*   **Breaking Change:** Removing or renaming fields or types clients already use.
*   **Schema Version:** Logical version tracked via registry, tags, or deploy metadata.
*   **Consumer Contract:** The set of operations clients run against your graph.
*   **Compatibility Check:** Automated diff flagging dangerous schema edits.

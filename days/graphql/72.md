# Day 72: Security Hardening - Rate Limiting and Abuse Prevention

## 📝 Today's Tasks

1.  **Identify Abuse Vectors:** List expensive queries, brute-force mutations, and batch attacks.
2.  **Apply IP Rate Limits:** Throttle requests per IP at the HTTP gateway or middleware layer.
3.  **Per-Token Limits:** Apply stricter limits for authenticated users abusing expensive fields.
4.  **Return 429 Responses:** Use standard rate-limit headers and clear error messages.

## 📖 Task Explanations

GraphQL's single endpoint simplifies clients but concentrates abuse. Rate limiting often happens outside the executor—at API gateways or Express middleware—before queries consume database resources.

Combine rate limits with complexity analysis from Day 71 for defense in depth.

## 💡 What You'll Gain

By the end of today, you will implement rate limiting strategies protecting your GraphQL endpoint from abuse.

## 📚 Dictionary for Developers

*   **Rate Limiting:** Restricting request frequency per client identity or IP.
*   **429 Too Many Requests:** HTTP status indicating the client exceeded allowed throughput.
*   **Abuse Vector:** A pattern attackers use to overload or scrape an API.
*   **Throttle:** Delaying or rejecting requests after a threshold is exceeded.
*   **Gateway:** Edge layer enforcing policies before traffic reaches the GraphQL server.

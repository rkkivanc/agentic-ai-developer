# Day 16: Mutations & Inputs - Mutation Basics and Side Effects

## 📝 Today's Tasks

1.  **Define Mutation Root Fields:** Add write operations to your Mutation type with clear, verb-based names.
2.  **Understand Side Effects:** Recognize that mutations change server state unlike queries.
3.  **Run Create and Delete Mutations:** Execute mutations in a sandbox and inspect returned data.
4.  **Separate Reads and Writes:** Keep expensive fetches as queries; use mutations only for changes.

## 📖 Task Explanations

Mutations are GraphQL's write path. They should be named for business actions—`createPost`, `updateProfile`—not generic CRUD unless that matches your domain.

Although GraphQL allows querying within mutations, best practice is to return only what the client needs to update its cache or UI.

## 💡 What You'll Gain

By the end of today, you will define and execute mutations that perform create, update, and delete operations.

## 📚 Dictionary for Developers

*   **Input Object:** A structured input type grouping multiple arguments for mutations.
*   **Side Effect:** A state change on the server caused by a mutation operation.
*   **Payload Type:** A return type wrapping the mutated entity plus metadata like errors or status.
*   **Idempotency:** Property where repeating the same mutation produces the same final state.
*   **Write Operation:** Any GraphQL mutation that creates, updates, or deletes server data.

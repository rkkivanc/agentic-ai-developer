# Day 82: Caching & Performance - Server-Side Caching Strategies

## 📝 Today's Tasks

1.  **Cache Resolver Results:** Memoize expensive pure resolvers within a request or short TTL.
2.  **HTTP Response Caching:** Set cache headers only on appropriate queries—never on personalized data.
3.  **Redis Layer:** Sketch caching user sessions or frequently read objects keyed by ID.
4.  **Invalidate on Mutation:** Delete or update cache entries when related mutations succeed.

## 📖 Task Explanations

Server caching must respect per-user authorization. Only cache responses or objects that are identical for all callers, or key caches by user ID.

Mutations should trigger invalidation—stale cached lists after `createPost` frustrate users and violate expectations.

## 💡 What You'll Gain

By the end of today, you will apply server-side caching with safe invalidation around mutations.

## 📚 Dictionary for Developers

*   **Resolver Memoization:** Caching a resolver result for the duration of one request.
*   **TTL (Time To Live):** How long a cached value remains valid before refresh.
*   **Cache Key:** A unique string identifying what is stored—often type plus ID.
*   **Invalidation:** Removing stale cache entries when underlying data changes.
*   **Personalized Data:** Responses that differ per user and must not be shared in public caches.

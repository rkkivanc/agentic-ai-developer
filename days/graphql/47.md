# Day 47: Clients & Frontend Integration - Queries, Loading States, and Cache

## 📝 Today's Tasks

1.  **Render Loading UI:** Use `loading` from `useQuery` or urql's `fetching` flag in components.
2.  **Handle Errors:** Display GraphQL and network errors distinctly for users.
3.  **Set Fetch Policies:** Try `cache-first`, `network-only`, and `cache-and-network` for the same query.
4.  **Refetch on Action:** Call `refetch` after a mutation or user refresh gesture.

## 📖 Task Explanations

Client integration is not just fetching—it's UX. GraphQL queries are asynchronous; users need loading and error feedback while the cache decides whether to hit the network.

Fetch policies trade freshness for speed. Understanding them prevents surprise stale data in production dashboards.

## 💡 What You'll Gain

By the end of today, you will render loading and error states and control Apollo Client or urql cache fetch behavior.

## 📚 Dictionary for Developers

*   **Fetch Policy:** Rules controlling whether the client reads cache, network, or both.
*   **cache-first:** Return cached data if present, otherwise fetch from network.
*   **Refetch:** Manually re-execute a query against the server.
*   **Network Error:** Transport failure distinct from GraphQL field errors.
*   **Loading State:** UI indication that a query has not yet completed.

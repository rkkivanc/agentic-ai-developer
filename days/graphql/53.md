# Day 53: Subscriptions & Realtime - Pub/Sub Backends for Subscriptions

## 📝 Today's Tasks

1.  **Pick a Pub/Sub:** Evaluate Redis pub/sub, MQTT, or in-memory EventEmitter for development.
2.  **Publish from Mutations:** Emit events when data changes so subscribers receive updates.
3.  **Wire AsyncIterator:** Connect GraphQL subscription resolvers to your pub/sub library.
4.  **Scale Note:** Document why in-memory pub/sub fails across multiple server instances.

## 📖 Task Explanations

GraphQL subscription resolvers typically return async iterables fed by a pub/sub bus. Mutations publish; subscription resolvers subscribe and forward events to clients.

Production needs a distributed pub/sub—Redis, Kafka, or a managed service—so every API instance receives events to fan out.

## 💡 What You'll Gain

By the end of today, you will connect GraphQL subscriptions to a pub/sub backend publishing mutation events.

## 📚 Dictionary for Developers

*   **Pub/Sub:** Publish-subscribe messaging decoupling event producers and consumers.
*   **AsyncIterator:** Object yielding events consumed by the subscription executor.
*   **Event Bus:** Infrastructure carrying events from publishers to subscribers.
*   **Fan-Out:** Delivering one published event to many subscribed clients.
*   **Redis Pub/Sub:** Common lightweight broker for GraphQL subscription demos.

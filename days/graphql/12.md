# Day 12: Queries & Selection Sets - Arguments and Field Parameters

## 📝 Today's Tasks

1.  **Add Field Arguments in SDL:** Define arguments on fields such as `user(id: ID!)` or `posts(limit: Int)`.
2.  **Pass Arguments in Queries:** Supply literal values and observe typed coercion in responses.
3.  **Use Default Values:** Set defaults in SDL for optional arguments like pagination limits.
4.  **Compare Argument vs Input Object:** Note when simple arguments suffice versus input types for mutations.

## 📖 Task Explanations

Field arguments let clients parameterize queries without creating new endpoints. Each argument has a type and optional default, enforced during validation.

Defaults reduce boilerplate for common cases while keeping the schema explicit about available parameters.

## 💡 What You'll Gain

By the end of today, you will define and use field arguments to filter, sort, and parameterize queries.

## 📚 Dictionary for Developers

*   **Argument:** A named parameter on a field, specified in the query and defined in the schema.
*   **Variable:** A dynamic value passed separately from the query document using a `$` prefix.
*   **Fragment:** A reusable selection set that can be spread into multiple operations.
*   **Alias:** A query-time rename of a field in the response using `alias: fieldName` syntax.
*   **Directive:** An annotation such as `@include` or `@skip` that modifies query execution.
*   **Operation Name:** An optional label for a query that aids debugging and server logs.

# Day 88: Performance - Concurrency Tuning

## 📝 Today's Tasks

1.  **Right-Size Worker Pools:** Match goroutine counts to CPU and I/O constraints.
2.  **Avoid Goroutine Leaks:** Ensure workers exit when contexts cancel.
3.  **Use errgroup:** Coordinate error propagation across concurrent tasks.
4.  **Profile Goroutines:** Inspect goroutine profiles for unexpected growth.

## 📖 Task Explanations

More goroutines is not always faster. CPU-bound work rarely benefits beyond GOMAXPROCS; I/O-bound work needs enough parallelism without unbounded spawning.

`errgroup` combines goroutines with error propagation—useful in startup and batch jobs. Goroutine leaks show up as memory growth and scheduling overhead.

Profile goroutine counts under load.

## 💡 What You'll Gain

By the end of today, you will tune concurrency with worker limits, errgroup, and goroutine profiling.

## 📚 Dictionary for Developers

*   **GOMAXPROCS:** Setting controlling how many OS threads execute Go code simultaneously.
*   **Worker Pool:** A fixed set of goroutines processing jobs from a queue.
*   **errgroup:** A package running goroutines with shared error propagation and cancellation.
*   **Goroutine Leak:** Goroutines that never exit, accumulating over time.
*   **I/O-bound:** Work dominated by waiting on network, disk, or other external systems.

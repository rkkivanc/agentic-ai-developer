# Day 34: First REST API Project (MVP) - Validation and Error Responses

## 📝 Today's Tasks

1.  **Validate Input:** Reject empty titles, overly long strings, and malformed IDs.
2.  **Map Errors to Status:** Return 400 for validation, 404 for missing IDs, 500 only for true internals.
3.  **Centralize Error Writing:** Create a helper to write JSON problem responses consistently.
4.  **Test Unhappy Paths:** Add handler tests for each failure type.

## 📖 Task Explanations

Validation is part of API design, not an afterthought. Clients should receive actionable 400 responses with clear messages.

Reserve 500 for unexpected internal failures—never for bad user input. Central helpers keep error JSON uniform across handlers.

Tests for unhappy paths protect you when refactoring storage or routing later.

## 💡 What You'll Gain

By the end of today, you will validate inputs and return consistent, well-tested HTTP error responses.

## 📚 Dictionary for Developers

*   **400 Bad Request:** HTTP status for client-side validation or malformed request errors.
*   **Problem Response:** A JSON body describing an error code, message, and optional details.
*   **Internal Error:** An unexpected server failure, usually logged and returned as 500.
*   **Validation Rule:** A constraint on input such as required fields, length limits, or formats.
*   **Defensive Handler:** A handler that assumes clients may send bad data and responds safely.

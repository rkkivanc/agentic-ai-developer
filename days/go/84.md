# Day 84: Caching & Messaging - Event-Driven Patterns

## 📝 Today's Tasks

1.  **Outbox Pattern Overview:** Reliably publish events after database commits.
2.  **Idempotent Handlers:** Use deduplication keys to handle redelivered messages.
3.  **Dead Letter Queue:** Route poison messages after max retries for inspection.
4.  **Saga Sketch:** Outline multi-step workflows coordinated by events at a high level.

## 📖 Task Explanations

Event-driven designs trade synchronous simplicity for scalability and resilience. The outbox pattern prevents lost events when DB commits succeed but publish fails.

Idempotent handlers and dead-letter queues are production necessities. Sagas coordinate long workflows—sketch them even if you do not implement fully today.

Patterns matter more than any one broker API.

## 💡 What You'll Gain

By the end of today, you will understand outbox, idempotent consumers, and dead-letter handling patterns.

## 📚 Dictionary for Developers

*   **Outbox Pattern:** Storing outbound events in the database and publishing them asynchronously.
*   **Dead Letter Queue:** A queue holding messages that failed processing after retries.
*   **Deduplication Key:** An identifier ensuring the same logical event is not processed twice.
*   **Saga:** A sequence of local transactions coordinated via events across services.
*   **Poison Message:** A message that always fails processing and must be quarantined.

# Day 81: Caching & Messaging - In-Memory and Redis Caching

## 📝 Today's Tasks

1.  **Identify Cache Candidates:** Choose read-heavy endpoints safe to cache briefly.
2.  **Add Redis Client:** Connect to Redis and set/get JSON blobs with TTLs.
3.  **Cache-Aside Pattern:** Read cache first, on miss load DB and populate cache.
4.  **Invalidate on Write:** Delete or update cache keys when data changes.

## 📖 Task Explanations

Caching reduces database load and latency for hot reads. Redis is a common external cache; in-memory works for single-instance prototypes.

Cache-aside is simple: application manages cache explicitly. TTLs limit staleness; invalidation on writes keeps data fresher.

Only cache what you can tolerate being briefly stale.

## 💡 What You'll Gain

By the end of today, you will implement cache-aside reads and invalidation with Redis or in-memory cache.

## 📚 Dictionary for Developers

*   **Cache-Aside:** Application-managed caching where code reads cache before database.
*   **TTL (Time To Live):** Expiration duration after which a cache entry is discarded.
*   **Redis:** An in-memory data store often used for caching and pub/sub.
*   **Stale Data:** Cached values outdated relative to the database source of truth.
*   **Hot Path:** Frequently executed code or query benefiting most from caching.

# Day 59: Project Layout & Architecture - Domain Models and Services

## 📝 Today's Tasks

1.  **Rich Domain Types:** Use types that enforce invariants, not anemic data bags.
2.  **Validate in Domain:** Reject invalid state before persistence when possible.
3.  **Service Errors:** Return typed domain errors mapped to HTTP in transport layer.
4.  **Keep Handlers Thin:** Handlers should parse, call service, and render—nothing more.

## 📖 Task Explanations

Domain models encode business truth. Validation belongs near domain rules so invalid states never reach storage.

Services return domain errors; HTTP handlers map them to status codes. Thin handlers are easier to read and test.

This discipline pays off when APIs grow past a handful of endpoints.

## 💡 What You'll Gain

By the end of today, you will model domain rules in services and keep HTTP handlers thin translators.

## 📚 Dictionary for Developers

*   **Invariant:** A condition that must always hold for a domain object to be valid.
*   **Anemic Model:** A data struct with little behavior, often a sign logic leaked elsewhere.
*   **Domain Error:** An error representing a business failure, not a transport failure.
*   **Mapping Layer:** Code translating domain errors to HTTP or gRPC responses.
*   **Thin Handler:** HTTP code with minimal logic beyond parsing and rendering.

# Day 46: Databases (II) & Repositories - Repository Pattern

## 📝 Today's Tasks

1.  **Define Repository Interfaces:** Expose domain operations, not SQL strings, to higher layers.
2.  **Implement Postgres Repository:** Provide a concrete repository using `database/sql`.
3.  **Map Models to Domain:** Convert database models to API DTOs at boundaries.
4.  **Inject Repositories:** Pass repositories into handlers via constructors.

## 📖 Task Explanations

The repository pattern isolates persistence from business logic. Handlers should not know whether data lives in Postgres, SQLite, or memory.

Interfaces enable tests with fakes and swaps when storage changes. Mapping layers prevent leaking database tags into public JSON.

Constructor injection keeps dependencies explicit—no hidden globals.

## 💡 What You'll Gain

By the end of today, you will structure data access behind repository interfaces injected into handlers.

## 📚 Dictionary for Developers

*   **Repository Pattern:** An abstraction hiding storage implementation details from business logic.
*   **Domain Type:** A type representing business concepts rather than database layout.
*   **Mapping:** Translating between representations such as SQL rows and API DTOs.
*   **Fake Repository:** An in-memory implementation used in tests instead of a real database.
*   **Constructor Injection:** Supplying dependencies when creating a service or handler.

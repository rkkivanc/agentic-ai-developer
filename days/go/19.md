# Day 19: Concurrency Basics - sync Package Primitives

## 📝 Today's Tasks

1.  **Use Mutex:** Protect shared counters with `sync.Mutex` or `sync.RWMutex`.
2.  **Try Once:** Initialize expensive resources exactly once with `sync.Once`.
3.  **Use atomic:** Increment simple counters with `sync/atomic` when mutexes are heavier than needed.
4.  **Compare Approaches:** Summarize when channels versus mutexes are clearer for your use case.

## 📖 Task Explanations

Not every problem needs a channel. Mutexes and atomics are straightforward tools for protecting in-memory state when ownership is shared.

`sync.Once` is idiomatic for lazy initialization of databases, clients, and configuration. Atomics suit simple numeric stats; mutexes generalize to richer structures.

Go developers mix channels and mutexes pragmatically—choose the model that makes correctness obvious.

## 💡 What You'll Gain

By the end of today, you will protect shared state with mutexes, perform one-time initialization, and choose between channels and locks.

## 📚 Dictionary for Developers

*   **Mutex:** A mutual exclusion lock ensuring only one goroutine accesses critical state at a time.
*   **RWMutex:** A mutex that allows many readers or one writer, useful for read-heavy caches.
*   **sync.Once:** A type ensuring a function runs exactly once, even with concurrent callers.
*   **atomic:** Low-level operations on integers and pointers without explicit mutex locking.
*   **Critical Section:** A region of code that must not run concurrently on shared data without synchronization.

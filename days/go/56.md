# Day 56: Project Layout & Architecture - Standard Go Project Layout

## 📝 Today's Tasks

1.  **Organize cmd and internal:** Place binaries under `cmd/` and private code under `internal/`.
2.  **Separate internal packages:** Split `httpapi`, `service`, `repository`, and `domain` packages.
3.  **Keep main Thin:** Wire dependencies in `main` without business logic.
4.  **Add README Structure:** Document package responsibilities for newcomers.

## 📖 Task Explanations

Go has community conventions that make repositories familiar. The standard layout is not official law, but it helps teammates navigate quickly.

Thin `main` functions wire the world; they do not implement business rules. Package names should describe roles, not generic `util` junk drawers.

Document structure in README so onboarding takes hours, not days.

## 💡 What You'll Gain

By the end of today, you will organize a multi-package Go service using conventional layout.

## 📚 Dictionary for Developers

*   **cmd/:** Directory containing main packages that compile to binaries.
*   **internal/:** Packages private to your module, enforced by the Go compiler.
*   **Wire:** Connecting concrete implementations to interfaces at startup.
*   **Package Coupling:** How much packages depend on each other; lower is usually better.
*   **Onboarding:** How quickly a new developer becomes productive in a codebase.

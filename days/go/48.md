# Day 48: Databases (II) & Repositories - Testing with Databases

## 📝 Today's Tasks

1.  **Use Test Database:** Run tests against a disposable database or schema.
2.  **Reset State:** Truncate tables or use transactions rolled back after each test.
3.  **Test Repositories:** Write integration tests covering create, read, and not-found paths.
4.  **Parallel Safety:** Avoid parallel tests sharing one dirty schema without isolation.

## 📖 Task Explanations

Database code deserves integration tests. Unit tests with fakes are not enough to catch SQL typos or constraint violations.

Use a dedicated test database, reset state between tests, and keep tests independent. Parallelism is great only when schemas do not collide.

These tests cost more than unit tests but pay for themselves quickly.

## 💡 What You'll Gain

By the end of today, you will integration-test repositories against a real database safely.

## 📚 Dictionary for Developers

*   **Integration Test:** A test verifying multiple components together, such as repository plus database.
*   **Test Schema:** A separate database schema used only for tests.
*   **Truncate:** Removing all rows from tables to reset state quickly.
*   **Test Isolation:** Ensuring tests do not share mutable state that causes order-dependent failures.
*   **Constraint:** A database rule such as uniqueness or foreign keys enforced on writes.

# Day 72: Observability & Resilience - Metrics with Prometheus

## 📝 Today's Tasks

1.  **Expose /metrics:** Serve Prometheus metrics from your HTTP server.
2.  **Add Counters and Histograms:** Track request counts and latency distributions.
3.  **Label Carefully:** Avoid high-cardinality labels like raw user IDs.
4.  **Scrape Locally:** Point Prometheus or curl at `/metrics` to verify output.

## 📖 Task Explanations

Metrics summarize system health: traffic, errors, latency. Prometheus pull model fits Go services well.

Counters track totals; histograms approximate latency percentiles. Cardinality explosions break metrics systems—label routes, not unbounded IDs.

Verify `/metrics` locally before relying on dashboards.

## 💡 What You'll Gain

By the end of today, you will expose Prometheus metrics for requests and latency.

## 📚 Dictionary for Developers

*   **Prometheus:** A metrics collection system scraping HTTP endpoints periodically.
*   **Counter:** A metric only increasing, counting events like total requests.
*   **Histogram:** A metric sampling observations into buckets for percentile estimates.
*   **Cardinality:** The number of unique time series, exploded by too many label values.
*   **Scrape:** Prometheus fetching metrics from a target endpoint.

# Day 87: Performance - Memory and Allocation Tuning

## 📝 Today's Tasks

1.  **Reduce Allocations:** Reuse buffers with `sync.Pool` where profiling shows churn.
2.  **Preallocate Slices:** Set slice capacity when sizes are known ahead of time.
3.  **Avoid String Concat Loops:** Use `strings.Builder` for large string assembly.
4.  **Measure Allocs/op:** Track allocations per benchmark iteration.

## 📖 Task Explanations

Go's GC is good, but allocations still cost CPU. Profiles often show surprising string and slice churn.

`sync.Pool` helps reuse temporary objects—use sparingly and measure. Preallocation and builders are simpler first steps.

Watch `allocs/op` in benchmarks when tuning hot paths.

## 💡 What You'll Gain

By the end of today, you will reduce allocations in hot paths using data from benchmarks and profiles.

## 📚 Dictionary for Developers

*   **sync.Pool:** A pool of temporary objects subject to clearing between GC cycles.
*   **Allocation:** Reserving heap memory for new values, work the garbage collector must later reclaim.
*   **strings.Builder:** An efficient mutable string builder minimizing allocations.
*   **GC (Garbage Collector):** Go's automatic memory manager reclaiming unused heap objects.
*   **allocs/op:** Allocations per benchmark operation, reported with `-benchmem`.

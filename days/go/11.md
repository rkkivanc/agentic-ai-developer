# Day 11: Errors, Collections & I/O - Error Values and Handling

## 📝 Today's Tasks

1.  **Return Errors:** Write functions that return `(T, error)` and check errors at call sites.
2.  **Create Error Values:** Use `errors.New` and `fmt.Errorf` to construct meaningful error messages.
3.  **Wrap Errors:** Add context with `%w` in `fmt.Errorf` and inspect chains with `errors.Is` and `errors.As`.
4.  **Fail Fast:** Practice early returns instead of deep nesting when handling failures.

## 📖 Task Explanations

Go treats errors as values, not exceptions. This design keeps control flow explicit and encourages you to handle failures at every boundary.

You'll learn to return errors alongside results, wrap them with context, and inspect error chains without string matching. These patterns appear in virtually every production Go service.

Failing fast with early returns keeps handlers readable and prevents partially applied state when something goes wrong.

## 💡 What You'll Gain

By the end of today, you will handle failures idiomatically with explicit error returns, wrapping, and inspection helpers.

## 📚 Dictionary for Developers

*   **error:** A built-in interface type representing a failure condition. The zero value is nil (no error).
*   **Error Wrapping:** Attaching context to an error while preserving the original for inspection with `errors.Is`/`errors.As`.
*   **errors.Is:** A function that checks whether an error matches a target value in a wrapped error chain.
*   **errors.As:** A function that finds the first error in a chain that matches a target type.
*   **Fail Fast:** Returning immediately when an error occurs instead of continuing with invalid state.

# Day 33: First REST API Project (MVP) - In-Memory Storage Layer

## 📝 Today's Tasks

1.  **Create a Store Interface:** Define storage operations behind an interface for testability.
2.  **Implement Mutex Protection:** Guard an in-memory map with `sync.RWMutex` for concurrent safety.
3.  **Generate IDs:** Assign unique IDs on create using UUIDs or incremental counters.
4.  **Return Copies:** Avoid leaking internal map references by returning copies or DTOs.

## 📖 Task Explanations

Even in-memory stores deserve interfaces and concurrency control. Your HTTP server handles requests concurrently; an unprotected map will race.

Interfaces let you swap in database-backed repositories later without rewriting handlers. Returning copies prevents callers from mutating internal state accidentally.

This is your first repository-shaped layer—naming and boundaries matter.

## 💡 What You'll Gain

By the end of today, you will implement a thread-safe in-memory store behind a clean interface.

## 📚 Dictionary for Developers

*   **Store:** A component abstracting create/read/update/delete operations for a resource.
*   **Repository:** A data-access layer hiding storage details from business logic.
*   **UUID:** A universally unique identifier often used for resource IDs in APIs.
*   **RWMutex:** A lock allowing concurrent readers but exclusive writers.
*   **Leakage:** Accidentally exposing internal mutable state to external callers.

# Day 76: Containers & CI/CD - Dockerizing Go Apps

## 📝 Today's Tasks

1.  **Write a Dockerfile:** Build your API in a container image with a minimal base.
2.  **Use Multi-Stage Build:** Compile in a builder image and copy the binary to a small runtime image.
3.  **Set ENTRYPOINT:** Run the compiled binary as PID 1 with proper signals.
4.  **Run Locally:** Build and run the container mapping ports and env vars.

## 📖 Task Explanations

Containers package your service with its runtime expectations. Multi-stage builds keep images small—distroless or alpine bases reduce attack surface.

Go compiles to static binaries, which makes containerization pleasant. Ensure PID 1 handles signals or use an init wrapper if needed.

Build and run locally before CI does.

## 💡 What You'll Gain

By the end of today, you will containerize your Go API with a multi-stage Dockerfile.

## 📚 Dictionary for Developers

*   **Docker Image:** A packaged filesystem snapshot used to run containers.
*   **Multi-Stage Build:** A Dockerfile technique separating build and runtime stages to shrink images.
*   **ENTRYPOINT:** The default command executed when a container starts.
*   **Distroless:** Minimal container images containing only the application and runtime essentials.
*   **PID 1:** The first process in a container responsible for signal handling.

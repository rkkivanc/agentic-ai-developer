# Day 71: Observability & Resilience - Structured Logging

## 📝 Today's Tasks

1.  **Use slog or zap:** Emit JSON or key/value logs instead of printf debugging.
2.  **Add Request Fields:** Include request ID, method, path, and duration on each line.
3.  **Log Levels:** Use info, warn, and error consistently with guidance.
4.  **Avoid Secrets:** Never log passwords, tokens, or PII unnecessarily.

## 📖 Task Explanations

Structured logs are machine-parseable and searchable. Include consistent fields so log aggregators can filter by request ID or user.

Log levels communicate severity—do not log everything as info. Secrets in logs become breaches.

Good logging is your first observability superpower.

## 💡 What You'll Gain

By the end of today, you will emit structured logs with request-scoped fields and sensible levels.

## 📚 Dictionary for Developers

*   **Structured Logging:** Logs as key/value or JSON records instead of unstructured text.
*   **slog:** Go 1.21+ standard library structured logging package.
*   **Log Level:** Severity such as DEBUG, INFO, WARN, ERROR controlling verbosity.
*   **PII:** Personally Identifiable Information that should not appear in logs casually.
*   **Correlation ID:** An identifier tying log lines to one request across components.

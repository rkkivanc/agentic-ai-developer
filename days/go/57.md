# Day 57: Project Layout & Architecture - Clean Architecture Layers

## 📝 Today's Tasks

1.  **Domain Layer:** Keep business rules free of HTTP and SQL details.
2.  **Service Layer:** Orchestrate use cases using repositories and domain types.
3.  **Transport Layer:** Map HTTP requests to service calls and responses.
4.  **Dependency Rule:** Ensure inner layers do not import outer layers.

## 📖 Task Explanations

Clean architecture separates concerns so business rules survive UI or database changes. Domain logic should not import `net/http` or SQL drivers.

Services implement use cases: create order, cancel subscription. Handlers translate HTTP to service calls and back.

The dependency rule keeps inner circles independent—test business logic without spinning up HTTP.

## 💡 What You'll Gain

By the end of today, you will separate domain, service, and transport layers with correct dependency direction.

## 📚 Dictionary for Developers

*   **Clean Architecture:** A layered design keeping business rules independent of frameworks and IO.
*   **Use Case:** An application-specific operation such as registering a user.
*   **Transport Layer:** Code handling external protocols like HTTP or gRPC.
*   **Dependency Rule:** Dependencies point inward; inner layers know nothing about outer layers.
*   **Business Rule:** A constraint or operation reflecting domain requirements, not HTTP details.

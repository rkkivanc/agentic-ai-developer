# Day 22: Testing Fundamentals - Table-Driven Tests

## 📝 Today's Tasks

1.  **Build Test Tables:** Define slices of inputs and expected outputs for multiple scenarios.
2.  **Use t.Run:** Create subtests with names for each table row to improve failure messages.
3.  **Cover Edge Cases:** Include zero values, empty strings, and error paths in your tables.
4.  **Keep Tables Readable:** Extract setup helpers when tables grow large.

## 📖 Task Explanations

Table-driven tests are the idiomatic Go pattern for exhaustive cases. One test function loops over many inputs, reducing duplication and encouraging new cases.

Subtests give each row a name, so failures point to the exact scenario. Include edge cases now—empty inputs, boundary values, and invalid parameters.

Readable tables are maintainable tables. Extract helpers when setup repeats, but keep expectations visible.

## 💡 What You'll Gain

By the end of today, you will write clear table-driven tests with subtests that cover happy and unhappy paths.

## 📚 Dictionary for Developers

*   **Table-Driven Test:** A single test function iterating over a table of cases with inputs and expected outputs.
*   **Subtest:** A nested test run via `t.Run`, shown separately in `go test` output.
*   **Test Case:** One row in a table describing inputs, expected results, and sometimes setup.
*   **Regression:** A bug reintroduced after being fixed; tests help prevent regressions.
*   **Test Helper:** A function marked with `t.Helper()` that centralizes setup without cluttering test bodies.

# Day 18: Generics - Constraints with extends

## 📝 Today's Tasks

1.  **Constrain a Generic:** Write `<T extends { id: string }>` so only objects with `id` are accepted.
2.  **Use Multiple Constraints:** Combine constraints with intersection types on the generic bound.
3.  **Constrain to keyof:** Build a `getProperty<T, K extends keyof T>(obj: T, key: K)` helper.
4.  **Compare Constrained vs any:** Refactor an unconstrained generic to a bounded one and note improved errors.

## 📖 Task Explanations

Unconstrained generics accept anything—including types that lack properties your function uses. Constraints (`T extends SomeShape`) tell the compiler what minimum shape `T` must have, unlocking safe property access inside the function body.

The `keyof` constraint pattern is everywhere: `pick`, `pluck`, and typed `Object.keys` wrappers all rely on `K extends keyof T` so returned values match the requested key. Better error messages are a hidden benefit—constraints fail at the call site with clear expectations instead of deep internal errors.

Reach for constraints when your function accesses specific properties or methods on `T`. Leave generics unbounded when the algorithm truly works for all types.

## 💡 What You'll Gain

By the end of today, you will constrain generics with `extends`, combine bounds, and write `keyof`-safe property accessors—the building blocks of typed utility libraries.

## 📚 Dictionary for Developers

*   **Generic Constraint:** An `extends` bound limiting which types may substitute for a type parameter.
*   **`keyof`:** An operator producing a union of an object's property names.
*   **Bounded Generic:** A type parameter restricted to types matching a minimum shape.
*   **Call-Site Error:** A compiler error at the invocation rather than inside function internals.
*   **Type-Safe Property Access:** Accessing `obj[key]` where `key` is proven valid for `obj`.

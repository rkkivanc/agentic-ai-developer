# Day 82: Full-stack Contracts & Shared Types - Contract-First API Design

## 📝 Today's Tasks

1.  **Define API Contract:** Document endpoints, request bodies, and responses as typed interfaces.
2.  **Share Zod Schemas:** Place request/response schemas in the shared package.
3.  **Generate Client Types:** Build a typed fetch client from shared response types.
4.  **Version the Contract:** Discuss how API versioning interacts with shared type semver.

## 📖 Task Explanations

Contract-first design defines the API before implementation. Shared Zod schemas validate server input and parse client responses with the same rules—disagreements become compile errors, not production bugs.

Typed clients consume shared response types—`getUser(): Promise<UserResponse>` imports `UserResponse` from shared package. When the server adds a required field, client code fails typecheck until updated.

Versioning shared contracts requires semver discipline—coordinate releases across apps.

## 💡 What You'll Gain

By the end of today, you will design a contract-first API with shared schemas and a typed client consuming the same types as the server.

## 📚 Dictionary for Developers

*   **Contract-First:** Defining API shapes before implementing handlers.
*   **Shared Schema:** Zod definitions consumed by both client and server.
*   **Typed Client:** HTTP wrapper using shared response types.
*   **API Versioning:** Managing breaking API changes across deployed services.
*   **Cross-App Typecheck:** Both frontend and backend failing build on contract drift.

# Day 87: Performance & Modern TS Features - const Assertions Deep Dive

## 📝 Today's Tasks

1.  **Apply as const:** Freeze arrays and objects to readonly literal types.
2.  **Build Tuple Types:** Use `as const` for fixed-length argument lists and discriminated unions.
3.  **Combine with satisfies:** Use `as const satisfies` for validated, literal-preserved configs.
4.  **Export Const Assertions:** Apply in shared packages for route and permission constants.

## 📖 Task Explanations

`as const` makes values deeply readonly with literal types—`['admin', 'user']` becomes `readonly ['admin', 'user']`, not `string[]`. Tuple inference enables discriminated unions built from const arrays.

`as const satisfies` combines validation and literal preservation—the modern pattern for config objects. Shared packages export `as const` objects so consumers get literal autocomplete and exhaustiveness checking.

These features reduce magic strings and widen bugs across large teams.

## 💡 What You'll Gain

By the end of today, you will master `as const`, combine it with `satisfies`, and export literal-preserved constants from shared modules.

## 📚 Dictionary for Developers

*   **`as const`:** Assertion making values deeply readonly with literal types.
*   **Readonly Tuple:** A fixed-length array with literal element types.
*   **`as const satisfies`:** Combining literal preservation with type validation.
*   **Constant Export:** Shared readonly literals consumed across packages.
*   **Literal Union from const:** Deriving union types from `as const` arrays via indexing.

# Day 92: Team Practices & Migration - JavaScript to TypeScript Migration

## 📝 Today's Tasks

1.  **Plan Incremental Migration:** Enable `allowJs` and `checkJs` selectively in a JS codebase.
2.  **Rename Strategically:** Migrate high-value modules first (`.js` → `.ts`) with typed boundaries.
3.  **Avoid Big Bang:** Document a file-by-file or package-by-package migration strategy.
4.  **Measure Progress:** Track `% typed` and remaining `any` count as migration metrics.

## 📖 Task Explanations

Most teams inherit JavaScript. Incremental migration via `allowJs` lets TS and JS coexist—new files are `.ts`, old files convert gradually. `checkJs` with JSDoc types warms up JS files before rename.

Migrate boundaries first—API clients, parsers, shared utils—where types pay most. Avoid stopping all feature work for a big-bang rewrite. Track metrics to celebrate progress and justify investment.

Migration is a product decision—communicate timeline and risks to stakeholders.

## 💡 What You'll Gain

By the end of today, you will plan an incremental JS→TS migration with metrics, priorities, and coexistence configuration.

## 📚 Dictionary for Developers

*   **Incremental Migration:** Converting a codebase to TypeScript gradually, not all at once.
*   **`allowJs`:** Permitting JavaScript files in a TypeScript project.
*   **`checkJs`:** Type-checking JavaScript files using JSDoc annotations.
*   **Migration Metrics:** Tracking typed file percentage and remaining `any` usage.
*   **Big Bang Rewrite:** Risky full-codebase conversion—usually avoided.

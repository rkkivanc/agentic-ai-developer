# Day 26: Modules, Tooling & tsconfig - ES Modules in TypeScript

## 📝 Today's Tasks

1.  **Configure module and moduleResolution:** Set `module` and `moduleResolution` in `tsconfig.json` for your target environment.
2.  **Write ESM Imports/Exports:** Use `import`/`export` syntax and avoid CommonJS `require` in new code.
3.  **Use Type-Only Imports:** Apply `import type` to strip type imports from emitted JavaScript.
4.  **Resolve Extension Rules:** Understand when `.js` extensions are required in import paths for Node ESM.

## 📖 Task Explanations

Modules are how TypeScript projects scale beyond a single file. ES modules (`import`/`export`) are the modern standard for browsers and Node. Your `tsconfig.json` `module` setting controls what syntax `tsc` emits; `moduleResolution` controls how the compiler finds dependencies.

`import type` ensures type-only imports are erased completely—important for bundlers and `verbatimModuleSyntax`. Node ESM often requires explicit `.js` extensions in import paths even when source files are `.ts`—a common stumbling block worth confronting early.

Today connects your typed code to the wider JavaScript ecosystem. Misconfigured modules cause mysterious runtime errors; correct configuration makes imports predictable.

## 💡 What You'll Gain

By the end of today, you will configure ESM correctly, use type-only imports, and understand extension rules for Node—eliminating a major class of module-resolution bugs.

## 📚 Dictionary for Developers

*   **ES Module (ESM):** The standard `import`/`export` module system in modern JavaScript.
*   **`module` (tsconfig):** Compiler option controlling emitted module format.
*   **`moduleResolution`:** How TypeScript locates imported modules (`node10`, `bundler`, `nodenext`).
*   **`import type`:** An import used only for types, erased from emitted JavaScript.
*   **Type-Only Export:** `export type` exporting types without runtime values.

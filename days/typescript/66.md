# Day 66: Architecture & Patterns - Layered Architecture Basics

## 📝 Today's Tasks

1.  **Identify Layers:** Map presentation, application, domain, and infrastructure in a sample app.
2.  **Define Boundaries:** Document what each layer may import from which other layers.
3.  **Move Logic Down:** Push business rules out of handlers into domain/application services.
4.  **Draw Dependencies:** Sketch arrows showing allowed dependency direction (inward).

## 📖 Task Explanations

Architecture is about managing change cost. Layered architecture separates concerns: handlers parse HTTP, services orchestrate use cases, domain holds business rules, infrastructure talks to databases and APIs. Dependencies point inward—domain never imports HTTP frameworks.

TypeScript does not enforce layers; discipline and reviews do. Clear folder structure and `import` lint rules help. Types flow across layers as DTOs, domain models, and view models—never raw `any` from the database to the response.

Today builds vocabulary for Days 67–70 and your capstone.

## 💡 What You'll Gain

By the end of today, you will identify architectural layers, enforce dependency direction, and relocate business logic to appropriate modules.

## 📚 Dictionary for Developers

*   **Layered Architecture:** Organizing code into presentation, application, domain, and infrastructure.
*   **Dependency Rule:** Inner layers must not depend on outer layers.
*   **Domain Layer:** Core business rules independent of frameworks.
*   **Infrastructure Layer:** Database, HTTP clients, filesystem—external world adapters.
*   **Use Case:** An application service orchestrating a single business operation.

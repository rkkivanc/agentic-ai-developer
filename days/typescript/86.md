# Day 86: Performance & Modern TS Features - satisfies Operator

## 📝 Today's Tasks

1.  **Use satisfies:** Type-check object literals against a type while preserving literal inference.
2.  **Compare to Annotation:** Contrast `const x: Type = ...` vs `const x = ... satisfies Type`.
3.  **Build Config Maps:** Type theme tokens or route configs with `satisfies` for autocomplete plus safety.
4.  **Catch Excess Properties:** Observe how satisfies rejects unknown keys like explicit annotations.

## 📖 Task Explanations

The `satisfies` operator (TS 4.9+) validates a value against a type without widening literals to the annotated type. Config objects keep literal keys for autocomplete while still being checked against a broad interface—best of both worlds.

`const routes: Record<string, Route> = { home: ... }` loses literal keys; `const routes = { home: ... } satisfies Record<string, Route>` keeps them. This pattern appears in design tokens, routing tables, and lookup maps throughout modern codebases.

`satisfies` does not replace validation at boundaries—it improves internal config typing.

## 💡 What You'll Gain

By the end of today, you will use `satisfies` for config maps and understand when it beats type annotations for preserving literal types.

## 📚 Dictionary for Developers

*   **`satisfies`:** Operator validating a value against a type without widening inferred literals.
*   **Literal Preservation:** Keeping specific string/number literals in inferred types.
*   **Config Map:** An object mapping keys to configuration values.
*   **Excess Property Check:** Compiler rejecting unknown keys in object literals.
*   **Autocomplete Preservation:** Keeping narrow keys for editor suggestions after type checking.

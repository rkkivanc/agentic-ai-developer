# Day 10: Objects, Unions & Narrowing - Literal Types and Enums

## 📝 Today's Tasks

1.  **Use String and Numeric Literals:** Restrict values to specific constants like `'admin' | 'user'`.
2.  **Apply `as const`:** Create deeply readonly literal objects and tuples.
3.  **Compare `enum` vs Union of Literals:** Learn when each approach fits best.
4.  **Build a Typed Config Object:** Combine literals, unions, and `as const` for safe configuration.

## 📖 Task Explanations

Literal types shrink the set of allowed values to exact constants. Instead of any `string`, a role might be only `'admin' | 'guest'`. The `as const` assertion freezes an object so every property becomes a readonly literal—ideal for route maps, theme tokens, and lookup tables.

TypeScript `enum` generates runtime objects; union-of-literals is compile-time only. Many teams prefer literal unions for tree-shaking and simplicity, reserving `enum` for cases needing reverse lookups or numeric auto-increment. Understanding both lets you choose intentionally.

## 💡 What You'll Gain

By the end of today, you will express precise constant sets with literal types and `as const`, and choose between enums and literal unions for domain constants.

## 📚 Dictionary for Developers

*   **String Literal Type:** A type accepting only specific string values, e.g. `'pending' | 'done'`.
*   **`as const`:** An assertion making values deeply readonly with literal types.
*   **`enum`:** A TypeScript construct defining named constant members, with runtime representation.
*   **Constant Config:** A configuration object frozen with literal types to prevent invalid values.

# Day 94: Team Practices & Migration - ADRs and TypeScript Documentation

## 📝 Today's Tasks

1.  **Write an ADR:** Document a decision (e.g., Zod vs io-ts, enum vs union) in Architecture Decision Record format.
2.  **Document Type Conventions:** Create `TYPES.md` explaining interface vs type, naming, and export rules.
3.  **Onboard a Newcomer:** Write a 15-minute onboarding guide for TypeScript setup in your repo.
4.  **Link Docs to Code:** Reference ADRs and conventions in README and CONTRIBUTING.

## 📖 Task Explanations

Decisions without documentation get relitigated. ADRs capture context, options, and consequences—why you chose Zod, strict flags, or monorepo layout. Future teammates understand rationale without archaeology in Slack.

`TYPES.md` codifies conventions—how to name DTOs, where schemas live, when to use branded types. Onboarding docs reduce "how do I run this?" questions in PRs.

Documentation is part of shipping typed systems at team scale.

## 💡 What You'll Gain

By the end of today, you will write an ADR, document type conventions, and create onboarding material linked from project README.

## 📚 Dictionary for Developers

*   **ADR (Architecture Decision Record):** A document capturing a significant technical decision and its context.
*   **Convention Doc:** Written team standards for types, naming, and patterns.
*   **Onboarding Guide:** Short doc getting new contributors productive quickly.
*   **Decision Context:** The problem and constraints that motivated a technical choice.
*   **Knowledge Preservation:** Recording decisions so they survive team turnover.

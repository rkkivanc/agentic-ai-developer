# Day 71: Library & API Design - Public API Surface Design

## 📝 Today's Tasks

1.  **Audit Exports:** List every symbol consumers need vs internals to hide.
2.  **Minimize Surface Area:** Remove unnecessary exports and prefer composition over exposure.
3.  **Design for Discovery:** Order exports logically and name types for grep-friendly search.
4.  **Write API Guidelines:** Document principles for adding breaking vs non-breaking exports.

## 📖 Task Explanations

Library API design is product design. Every exported symbol is a promise—semver and consumer trust depend on stability. Smaller public surfaces are easier to maintain and harder to misuse. Hide helpers; export factories, types, and primary functions.

Names matter: `createUser` beats `init`; `UserCreateInput` beats `UCInput`. Consumers discover APIs via editor autocomplete and documentation—design for both.

Guidelines prevent accidental breaking changes—know when adding optional fields is safe vs when renaming is major.

## 💡 What You'll Gain

By the end of today, you will design a minimal, discoverable public API with documented guidelines for evolving exports safely.

## 📚 Dictionary for Developers

*   **Public API Surface:** All symbols exported for external consumption.
*   **Surface Minimization:** Exporting only what consumers genuinely need.
*   **Breaking Change:** A modification requiring consumers to update their code.
*   **Semantic Versioning:** Versioning scheme signaling compatibility (major.minor.patch).
*   **API Discoverability:** How easily consumers find and understand exports.

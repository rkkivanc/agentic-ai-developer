# Day 52: Validation & Boundaries (Zod) - Schema Design Patterns

## 📝 Today's Tasks

1.  **Compose Schemas:** Build nested objects and arrays with `z.array`, `z.union`, and `z.enum`.
2.  **Add Refinements:** Use `.refine` and `.superRefine` for cross-field validation rules.
3.  **Transform Data:** Apply `.transform` to normalize strings (trim, lowercase) during parse.
4.  **Default and Optional:** Model optional fields with `.optional()`, `.default()`, and `.nullable()`.

## 📖 Task Explanations

Real-world schemas are nested and conditional. Zod's combinators mirror TypeScript's unions and optionals while enforcing runtime rules. Refinements express constraints TypeScript cannot—like "end date after start date" or "password confirmation matches."

Transforms let parsers clean data at the boundary—trimming emails before storage. Defaults fill missing config values. Understanding optional vs nullable prevents the classic `undefined` vs `null` confusion at API edges.

Schema design is API design. Clear schemas produce clear errors for consumers and developers.

## 💡 What You'll Gain

By the end of today, you will compose complex Zod schemas with refinements, transforms, and correct optional/nullable modeling.

## 📚 Dictionary for Developers

*   **Schema Composition:** Combining Zod primitives into nested object and array schemas.
*   **`.refine`:** Adds custom validation rules to a schema.
*   **`.transform`:** Maps parsed values to a new shape during validation.
*   **`.optional()` / `.nullable()`:** Marking fields that may be absent or explicitly null.
*   **Validation Error:** Structured feedback when parsed data fails schema rules.

# Day 74: Library & API Design - Publishing and Semver for Types

## 📝 Today's Tasks

1.  **Configure package.json:** Set `name`, `version`, `exports`, `types`, and `files` for publishing.
2.  **Emit Declarations:** Enable `declaration` and `declarationMap` in tsconfig for consumers.
3.  **Document Breaking Changes:** Practice semver bumps for type signature changes.
4.  **Dry-Run Publish:** Run `npm pack` and inspect the tarball contents.

## 📖 Task Explanations

Publishing typed libraries means shipping JavaScript plus `.d.ts` files. `declaration: true` emits types; `declarationMap` helps consumers jump to source. The `exports` field routes types and JS correctly for dual ESM/CJS packages.

Type changes are API changes: making a parameter required is breaking; adding an optional field often is not. Follow semver honestly—consumers depend on your types as much as runtime behavior.

`npm pack` previews what ships—catch missing files before publishing.

## 💡 What You'll Gain

By the end of today, you will configure a publishable TypeScript package with declarations, correct exports, and semver awareness for type changes.

## 📚 Dictionary for Developers

*   **`declaration`:** tsconfig flag emitting `.d.ts` type files alongside JavaScript.
*   **`declarationMap`:** Source maps linking `.d.ts` back to `.ts` for editor navigation.
*   **`npm pack`:** Creates a tarball preview of what `npm publish` would ship.
*   **Type Breaking Change:** A type signature change forcing consumer code updates.
*   **Published Package:** An npm module consumed by other projects.

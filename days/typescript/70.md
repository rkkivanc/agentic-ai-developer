# Day 70: Architecture & Patterns - Practice

## 📝 Today's Tasks

1.  **Refactor Exercise:** Take a monolithic script and split into handlers, services, and repositories.
2.  **Introduce Interfaces:** Depend on repository interfaces in services; swap in-memory impl in tests.
3.  **Error Boundary:** Centralize error-to-HTTP mapping in one middleware or handler wrapper.
4.  **Review:** Draw a module diagram and verify no layer imports from a higher layer.

## 📖 Task Explanations

Practice day for architecture. Refactoring a monolith into layers is the most transferable exercise—most codebases are born monolithic and must be split as they grow. Interface-based repositories let tests use in-memory stores while production uses databases.

Centralized error mapping prevents each handler from inventing error JSON shapes. The dependency diagram review catches backwards imports—services importing handlers is a red flag.

These patterns are not enterprise fluff—they keep solo projects maintainable six months later.

## 💡 What You'll Gain

By the end of today, you will have refactored a project into layered architecture with testable interfaces and centralized error handling.

## 📚 Dictionary for Developers

*   **Layered Refactor:** Splitting monolithic code into handlers, services, and data layers.
*   **Dependency Diagram:** A visual map of which modules import which.
*   **Backwards Import:** A lower layer importing from a higher layer—an architecture violation.
*   **In-Memory Repository:** A test double implementing repository interface without a database.
*   **Error Middleware:** Central code mapping errors to HTTP responses.

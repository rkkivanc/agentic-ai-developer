# Day 75: Library & API Design - Practice

## 📝 Today's Tasks

1.  **Build a Mini SDK:** Create a typed client library for a fake API with overloads and branded IDs.
2.  **Hide Internals:** Ensure only the client class and public types are exported.
3.  **Add Overloads:** Support `get(id)` and `get({ filter })` with distinct return types.
4.  **Pack and Document:** Run `npm pack`, write README with typed examples, and add TSDoc.

## 📖 Task Explanations

Practice day for library design. A mini SDK exercises public API discipline—consumers import `ApiClient`, not internal `parseResponse` helpers. Branded IDs from Day 53 prevent argument transposition bugs.

Overloads model flexible entry points without losing return type precision. README examples and TSDoc are part of the deliverable—libraries are judged by developer experience.

The packed tarball confirms you are not shipping test files or source secrets.

## 💡 What You'll Gain

By the end of today, you will have built, documented, and packed a typed mini SDK demonstrating professional library API design.

## 📚 Dictionary for Developers

*   **SDK (Software Development Kit):** A typed client library abstracting an API for consumers.
*   **Internal Helper:** Module-private functions not exported to consumers.
*   **Developer Experience (DX):** How pleasant it is to use your library's API.
*   **Tarball Inspection:** Reviewing `npm pack` output before publish.
*   **Typed Client:** A class or module wrapping HTTP with typed methods.

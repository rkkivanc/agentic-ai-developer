# Day 81: Full-stack Contracts & Shared Types - Shared Packages in Monorepos

## 📝 Today's Tasks

1.  **Create a Shared Package:** Add `packages/types` or `packages/shared` in a monorepo workspace.
2.  **Export Domain Types:** Share `User`, `ApiResponse`, and error types between client and server.
3.  **Configure Workspace Protocol:** Use `workspace:*` dependencies linking packages.
4.  **Build Order:** Ensure shared package builds before apps via project references or turborepo.

## 📖 Task Explanations

Full-stack TypeScript shines when types are shared, not duplicated. A `packages/shared` workspace exports domain types, Zod schemas, and constants consumed by frontend and backend. npm/pnpm workspaces with `workspace:*` link locally without publishing.

Build order matters—apps depend on compiled shared output. Turborepo or `tsc --build` orchestrate dependency graphs. One source of truth means API changes break both sides at compile time, not in production.

Monorepos are the default at scale; learning workspace patterns early prevents painful extraction later.

## 💡 What You'll Gain

By the end of today, you will create a shared types package in a monorepo and wire client/server apps to consume it.

## 📚 Dictionary for Developers

*   **Workspace Package:** A local monorepo package linked via workspace protocol.
*   **`workspace:*`:** Dependency specifier linking to local workspace packages.
*   **Shared Types Package:** A module exporting types and schemas used by multiple apps.
*   **Build Order:** Compiling dependencies before dependents in a monorepo.
*   **Single Source of Truth:** One canonical type definition consumed everywhere.

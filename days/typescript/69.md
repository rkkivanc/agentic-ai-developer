# Day 69: Architecture & Patterns - Module Boundaries and Error Boundaries

## 📝 Today's Tasks

1.  **Define Public Module API:** Export only intended symbols from each folder via barrel or index files.
2.  **Enforce with Lint:** Configure `import/no-restricted-paths` or similar to block cross-layer imports.
3.  **Centralize Error Mapping:** Map domain errors to HTTP/status codes in one boundary module.
4.  **Document Module Contracts:** Write brief README per module explaining its public API.

## 📖 Task Explanations

Module boundaries are architectural guardrails. Barrel files (`index.ts`) can hide internals—export only public types and functions. Lint rules blocking `services/` from importing `handlers/` turn architecture into enforced policy.

Error boundaries translate internal failures to external responses—`UserNotFound` becomes 404, `ValidationError` becomes 400. Domain code should not know HTTP; the boundary layer does.

Module READMEs cost minutes and save hours during onboarding.

## 💡 What You'll Gain

By the end of today, you will define explicit module public APIs, enforce import boundaries, and centralize error translation between layers.

## 📚 Dictionary for Developers

*   **Barrel File:** An `index.ts` re-exporting a module's public API.
*   **Module Boundary:** The edge defining what a folder exposes vs keeps private.
*   **Import Restriction:** Lint rules preventing architectural layer violations.
*   **Error Mapping:** Translating internal errors to external response formats.
*   **Public API:** The intentionally exported surface of a module.

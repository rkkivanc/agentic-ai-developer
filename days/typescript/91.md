# Day 91: Team Practices & Migration - Type-Focused Code Review

## 📝 Today's Tasks

1.  **Review Checklist:** Create a checklist covering types, boundaries, exports, and test coverage.
2.  **Review a PR:** Walk through a sample diff focusing on type safety, not just logic.
3.  **Spot Anti-Patterns:** Flag `any`, unsafe casts, missing validation, and floating promises.
4.  **Suggest Improvements:** Practice constructive comments proposing `unknown`, Zod, or Result types.

## 📖 Task Explanations

Code review is where team TypeScript culture is enforced. A type-focused checklist asks: Are boundaries validated? Are exports minimal? Do new public APIs have tests and TSDoc? Are async patterns correct?

Anti-patterns have recognizable smells—`as User` on JSON, `any` in exported functions, try/catch with bare `any`. Reviewers teach by suggesting concrete alternatives, not just blocking.

Good reviews balance velocity and safety—explain why a change matters, offer code snippets.

## 💡 What You'll Gain

By the end of today, you will conduct type-focused code reviews with a checklist, spotting anti-patterns and suggesting typed alternatives.

## 📚 Dictionary for Developers

*   **Review Checklist:** Structured items ensuring type safety in every PR.
*   **Type Anti-Pattern:** A habit that undermines TypeScript safety (e.g., `any`, blind `as`).
*   **Constructive Review:** Feedback explaining why and how to improve, not just rejecting.
*   **Export Review:** Checking new public APIs are intentional and documented.
*   **Team Culture:** Shared habits around types, testing, and review standards.

# Day 58: Testing (runtime + type tests) - expect-type and Type-Level Tests

## 📝 Today's Tasks

1.  **Install expect-type:** Add type tests asserting inferred types equal expected types.
2.  **Test Utility Types:** Assert `Pick<User, 'id'>` resolves correctly with `expectTypeOf`.
3.  **Catch Regressions:** Write type tests that fail if a generic inference changes.
4.  **Integrate in CI:** Run type tests as part of your test script or separate `typecheck` job.

## 📖 Task Explanations

Runtime tests do not catch type regressions—type tests do. `expect-type` (or `tsd`) asserts compile-time relationships: that a function returns `Promise<User>`, that a generic infers correctly, that a refactor did not widen types to `any`.

Library authors rely heavily on type tests; application teams benefit around utility types and complex generics. CI should run `tsc --noEmit` plus type tests—types are part of your contract.

Type tests document intent: "this helper must preserve keys" becomes an executable assertion.

## 💡 What You'll Gain

By the end of today, you will write type-level tests catching inference regressions and integrate type testing into your CI mindset.

## 📚 Dictionary for Developers

*   **expect-type:** A library asserting TypeScript types at compile time.
*   **Type Test:** A test verifying types, not runtime values.
*   **Type Regression:** An unintended change to inferred or exported types.
*   **`expectTypeOf`:** API for asserting properties of a type in tests.
*   **CI Typecheck:** Running `tsc --noEmit` and type tests in continuous integration.

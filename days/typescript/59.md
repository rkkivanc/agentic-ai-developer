# Day 59: Testing (runtime + type tests) - Integration Testing

## 📝 Today's Tasks

1.  **Test an API Route:** Spin up a server (or use supertest) and test HTTP request/response cycles.
2.  **Validate Response Shapes:** Combine runtime assertions with Zod schema parsing in tests.
3.  **Seed Test Data:** Create and tear down test fixtures for database or file-backed tests.
4.  **Separate Unit vs Integration:** Document which tests are fast-isolated vs slower end-to-end.

## 📖 Task Explanations

Integration tests verify components work together—HTTP handlers calling services, parsers reading real files. They are slower but catch wiring bugs unit tests miss. Tools like `supertest` send HTTP to your app without manual port juggling.

Assert response shapes by parsing through Zod—if the API contract drifts, tests fail at both runtime and schema level. Test data seeding must be isolated—parallel CI jobs cannot share dirty state.

Label tests clearly so developers know what failed and how to run subsets.

## 💡 What You'll Gain

By the end of today, you will write integration tests for HTTP workflows, validate responses with schemas, and distinguish unit from integration coverage.

## 📚 Dictionary for Developers

*   **Integration Test:** A test verifying multiple components working together.
*   **supertest:** An HTTP assertion library for testing Node servers.
*   **Test Seeding:** Creating known data state before integration tests run.
*   **Response Assertion:** Verifying HTTP status, headers, and body in tests.
*   **Test Pyramid:** Many unit tests, fewer integration tests, rare full E2E.

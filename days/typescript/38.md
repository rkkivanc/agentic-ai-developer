# Day 38: Async, Promises & Errors - Typed Errors and Result Patterns

## 📝 Today's Tasks

1.  **Define Error Types:** Create discriminated error unions (`NetworkError | ValidationError`).
2.  **Implement Result<T, E>:** Model success and failure without throwing for expected errors.
3.  **Type Catch Blocks:** Use `unknown` in catch and narrow with `instanceof` or guards.
4.  **Compare Throw vs Result:** Document when each approach fits in your project.

## 📖 Task Explanations

Thrown exceptions are invisible in function signatures—TypeScript cannot list what a function might throw. Result types (`{ ok: true, value: T } | { ok: false, error: E }`) make failure explicit in return types, popular in Rust-inspired TypeScript and functional teams.

Discriminated error unions enable exhaustive handling in `switch` statements. In `catch`, treat errors as `unknown`—not `any`—then narrow before accessing properties. This prevents assuming all thrown values are `Error` instances.

Neither pattern wins everywhere: throws suit truly exceptional failures; Results suit expected validation failures. Professional code picks intentionally.

## 💡 What You'll Gain

By the end of today, you will model errors with discriminated unions and Result types, handle `catch` safely with `unknown`, and choose throw vs Result deliberately.

## 📚 Dictionary for Developers

*   **Result Type:** A union representing either success (`T`) or failure (`E`) without throwing.
*   **Discriminated Error:** An error union tagged by a `kind` or `code` field.
*   **Exception Transparency:** The fact that thrown errors are not reflected in function signatures.
*   **`unknown` in catch:** Typing caught values as `unknown` before narrowing.
*   **Expected vs Unexpected Failure:** Validation errors (expected) vs out-of-memory (exceptional).

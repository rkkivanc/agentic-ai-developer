# Day 54: Validation & Boundaries (Zod) - DTO Mapping and API Contracts

## 📝 Today's Tasks

1.  **Define Input/Output Schemas:** Create separate Zod schemas for create, update, and response DTOs.
2.  **Map to Domain Types:** Write functions converting parsed DTOs to internal domain models.
3.  **Strip Unknown Keys:** Use `.strict()` or mapping to prevent unexpected fields passing through.
4.  **Align with OpenAPI:** Sketch how Zod schemas could generate or mirror API documentation.

## 📖 Task Explanations

APIs expose different shapes than internal domain models. Create schemas accept passwords; response schemas omit them. Zod schemas become the contract—consumers and servers agree on validated shapes.

Mapping functions stay pure and typed: `toDomain(parsed: CreateUserDto): User`. Strict schemas reject over-posting attacks sending unexpected fields. This connects to full-stack contracts on Days 81–85.

DTO mapping is where types meet security—what you accept and what you leak are both schema decisions.

## 💡 What You'll Gain

By the end of today, you will define input/output Zod schemas, map DTOs to domain types, and enforce strict API contracts at boundaries.

## 📚 Dictionary for Developers

*   **DTO Schema:** A Zod schema describing data crossing an API boundary.
*   **`.strict()`:** Rejects objects with keys not defined in the schema.
*   **Domain Mapping:** Converting validated DTOs to internal business types.
*   **Over-Posting:** Sending unexpected fields in requests—a security concern.
*   **API Contract:** The agreed request/response shapes between client and server.

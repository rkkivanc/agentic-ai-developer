# Day 72: Library & API Design - Function Overloads

## 📝 Today's Tasks

1.  **Write Overload Signatures:** Declare multiple call signatures for one function name.
2.  **Implement with Union:** Write a single implementation handling all overload cases.
3.  **Model Optional Args:** Use overloads for different arity behaviors with precise return types.
4.  **Compare vs Generics:** Decide when overloads beat a single generic signature.

## 📖 Task Explanations

Overloads let one function name behave differently based on arguments—`createElement(tag)` returns different types per tag string. Consumers see precise types; implementation handles the union internally.

Overload syntax requires overload declarations plus a compatible implementation signature—often broader than individual overloads. They shine when return types depend on literal argument values in ways generics cannot express.

Overuse creates maintenance burden—prefer generics when one signature suffices.

## 💡 What You'll Gain

By the end of today, you will write function overloads with typed implementations and choose between overloads and generics appropriately.

## 📚 Dictionary for Developers

*   **Overload Signature:** An additional declared call signature for a function.
*   **Implementation Signature:** The actual function body signature compatible with all overloads.
*   **Arity:** The number of arguments a function accepts.
*   **Call Signature:** The parameter and return types describing how to call a function.
*   **Overload Resolution:** Compiler choosing the best matching overload for a call.

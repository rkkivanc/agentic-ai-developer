# Day 21: Advanced Types (I) - Partial, Required, and Readonly

## 📝 Today's Tasks

1.  **Use `Partial<T>`:** Type a function that updates only some fields of a `User` object.
2.  **Use `Required<T>`:** Convert optional API fields to required domain models after validation.
3.  **Apply `Readonly<T>`:** Prevent mutation of configuration objects passed through your app.
4.  **Combine Utilities:** Nest `Partial<Readonly<User>>` and predict the resulting shape before verifying.

## 📖 Task Explanations

Utility types transform existing types instead of duplicating field lists. `Partial<User>` makes every property optional—ideal for PATCH endpoints and update forms. `Required<T>` does the opposite after you have confirmed all fields exist. `Readonly<T>` shallowly freezes properties, signaling immutability at the type level.

These utilities are implemented with mapped types under the hood—you will peek at that mechanism on Day 24. For now, treat them as standardized transforms every TypeScript developer recognizes. Composing them (`Partial<Readonly<T>>`) models real constraints: you may update some fields, but not mutate the object's readonly shell.

Duplicating types manually leads to drift when the source shape changes. Utilities keep a single source of truth.

## 💡 What You'll Gain

By the end of today, you will apply `Partial`, `Required`, and `Readonly` to real update, validation, and configuration scenarios without rewriting object shapes by hand.

## 📚 Dictionary for Developers

*   **Utility Type:** A built-in generic transform like `Partial<T>` operating on existing types.
*   **`Partial<T>`:** Makes all properties of `T` optional.
*   **`Required<T>`:** Makes all properties of `T` required.
*   **`Readonly<T>`:** Makes all properties of `T` readonly (shallow).
*   **Single Source of Truth:** One canonical type transformed elsewhere, avoiding duplicate definitions.

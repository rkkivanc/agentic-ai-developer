# Day 53: Validation & Boundaries (Zod) - Branded Types and Parse-Don't-Validate

## 📝 Today's Tasks

1.  **Create Branded Types:** Use `z.brand` or intersection branding for `UserId` and `Email`.
2.  **Parse at Boundaries:** Validate once at entry points; pass typed data internally.
3.  **Reject Double Validation:** Avoid re-parsing the same object in every function.
4.  **Map Invalid to Errors:** Convert Zod issues into user-friendly error messages.

## 📖 Task Explanations

Branded types add nominal flavor to structural TypeScript—a `UserId` is a string you cannot confuse with `OrderId` after parsing. Zod branding ensures only validated values receive the brand.

"Parse, don't validate" means transform `unknown` into typed domain objects at boundaries, then trust types internally. Re-validating everywhere wastes CPU and obscures architecture—boundaries should be obvious.

Formatting Zod `issues` into field-level messages is standard API and form work—connect schema paths to UI field names.

## 💡 What You'll Gain

By the end of today, you will use branded types for domain primitives, parse at clear boundaries, and format validation errors for consumers.

## 📚 Dictionary for Developers

*   **Branded Type:** A type tagged to prevent mixing structurally identical values.
*   **Parse, Don't Validate:** Parse `unknown` into typed objects once at boundaries.
*   **Domain Primitive:** A typed wrapper around a primitive (Email, UserId).
*   **Zod Issue:** A structured validation error with path and message.
*   **Boundary Layer:** Code responsible for admitting only valid external data.

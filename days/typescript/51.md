# Day 51: Validation & Boundaries (Zod) - Introduction to Schema Validation

## 📝 Today's Tasks

1.  **Install Zod:** Add Zod to a project and define a `z.object` schema for a `User`.
2.  **Parse and Infer Types:** Use `z.infer<typeof UserSchema>` to derive TypeScript types from schemas.
3.  **Handle safeParse:** Compare `.parse()` (throws) vs `.safeParse()` (returns Result-like object).
4.  **Validate Unknown Input:** Parse `unknown` API data through a schema before use.

## 📖 Task Explanations

Runtime data—JSON from APIs, form submissions, env vars—is `unknown` until validated. Zod bridges runtime and compile time: schemas validate at runtime, and `z.infer` produces matching TypeScript types. One definition, two layers of safety.

`safeParse` returns `{ success, data/error }`—fitting Result patterns from Day 38. `.parse()` throws on failure—acceptable at trust boundaries where failure is exceptional.

This week replaces hope-based typing (`as User`) with proof-based typing (parsed schemas).

## 💡 What You'll Gain

By the end of today, you will define Zod schemas, infer TypeScript types from them, and validate unknown data at runtime with safe parsing.

## 📚 Dictionary for Developers

*   **Zod:** A TypeScript-first schema validation library.
*   **`z.infer`:** Extracts a TypeScript type from a Zod schema definition.
*   **`safeParse`:** Validates without throwing, returning success or error details.
*   **Runtime Validation:** Checking data shape at execution time, not just compile time.
*   **Trust Boundary:** Any point where external data enters your system.

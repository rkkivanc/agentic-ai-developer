# Day 83: Full-stack Contracts & Shared Types - OpenAPI and Codegen

## 📝 Today's Tasks

1.  **Generate OpenAPI from Zod:** Use tooling (e.g., zod-to-openapi) to produce an OpenAPI spec.
2.  **Or Consume OpenAPI:** Generate TypeScript types from an existing OpenAPI YAML with openapi-typescript.
3.  **Compare Approaches:** Document trade-offs of schema-first vs spec-first workflows.
4.  **Validate Against Spec:** Add a CI step verifying implementation matches OpenAPI.

## 📖 Task Explanations

OpenAPI is the lingua franca for HTTP APIs. Generating specs from Zod keeps code and docs synchronized. Conversely, `openapi-typescript` generates types from specs when backend is not TypeScript or contract is owned separately.

Codegen saves manual typing but can produce verbose types—wrapping generated types in domain aliases improves ergonomics. CI validation ensures handlers match documented paths and status codes.

Choose workflow based on who owns the contract—your team (Zod-first) or external partners (OpenAPI-first).

## 💡 What You'll Gain

By the end of today, you will generate or consume OpenAPI specs for typed clients and understand schema-first vs spec-first trade-offs.

## 📚 Dictionary for Developers

*   **OpenAPI:** A standard specification format describing HTTP APIs.
*   **Codegen:** Generating TypeScript types or clients from machine-readable specs.
*   **zod-to-openapi:** Tooling deriving OpenAPI documents from Zod schemas.
*   **openapi-typescript:** Generates TypeScript types from OpenAPI specifications.
*   **Spec Drift:** When implementation diverges from documented API contract.

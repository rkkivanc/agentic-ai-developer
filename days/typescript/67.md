2.  **Implement Service:** Build `CreateUserService` depending on the repository interface.
3.  **Swap Implementations:** Provide in-memory and (stub) database repository implementations.
4.  **Type Transactions:** Sketch how a `UnitOfWork` interface would coordinate commits.

## 📖 Task Explanations

The repository pattern hides data access behind a typed interface—services ask for `findById`, not SQL. This enables testing with in-memory stores and swapping databases without rewriting business logic.

Services orchestrate one use case: validate input, call repositories, emit events, return DTOs. They should not know about HTTP status codes—that is handler work.

Unit of Work coordinates multiple repository operations in one transaction—a pattern to recognize even if you defer full implementation.

## 💡 What You'll Gain

## 📚 Dictionary for Developers

*   **Repository:** An abstraction over data persistence with typed methods.
*   **Service (Application):** Orchestrates a use case using repositories and domain logic.
*   **CRUD:** Create, Read, Update, Delete operations on entities.
*   **Swappable Implementation:** Multiple concrete classes fulfilling one interface.
*   **Unit of Work:** Coordinates multiple writes in a single transactional boundary.
